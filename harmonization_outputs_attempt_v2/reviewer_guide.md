# Reviewer guide — CIMAC clinical harmonization outputs

A practical guide to the files in `harmonization_outputs/`. Written for
clinical / data reviewers who need to decide whether the harmonized
dataset is usable, and what remaining rule-level questions need a human
answer.

---

## 1. Executive summary

The pipeline harmonizes clinical data from **11 CIMAC trials** into a
single, consistent, reviewable table.

- **Primary deliverable:** `harmonized_11trials.csv` — 2,000 rows, 19
  columns, exactly 11 trials.
- **Excluded:** BACCI. It is not part of the CIMAC trial set; its source
  directory has been removed and the pipeline does not produce
  `harmonized_12trials.csv`. The `exclusion_and_order_checks.txt` file
  confirms zero BACCI rows in every artifact.
- **9-trial fidelity check:** `harmonized_9trials_reproduced.csv` is a
  re-derivation of the original 9-trial template from raw source files.
  Its rows have been **sorted to match the row order of
  `cross_trial_analysis_egk_april30_meta_9trials.csv`** so reviewers can
  diff the two files directly. `row_order_diagnostics.csv` confirms
  perfect 1:1 alignment (0 missing, 0 extra, 0 duplicate keys).
- **Uncertainty policy:** any value the pipeline cannot derive with
  high confidence is set to `NA` in the harmonized output and recorded
  in `flagged_for_review.csv` with the source columns it consulted and
  a question for the reviewer. The pipeline **never silently guesses**.
- **Provenance:** every harmonized cell can be traced back to its
  source file + column + row index via `provenance_long.csv` (~38,000
  rows).
- **Reproducibility:** the entire output set is produced by a single
  command, `./scripts/run_full_harmonization.sh`. No file in
  `harmonization_outputs/` should ever be hand-edited.

## 2. File-by-file overview

Categories used below: **primary** (harmonized data), **validation/audit**,
**provenance**, **human-review**, **diagnostic/log**.

### Primary dataset

#### `harmonized_11trials.csv` *(primary)*
- Format: CSV, 2,000 rows × 19 cols.
- Generated by: `scripts/extract_harmonized_clinical.py`.
- What you get: the final 11-trial harmonized table. One row per
  (trial, cimac_part_id, Cimac.id, Collection_Event). 19 columns include
  demographics, arm, treatment, response, survival, and the derived
  `bor_bin` / `pfs_bin` fields (which are policy-NA — see decisions
  D1–D2 below).
- When to use: **last** — after you have reviewed rule-level open items.

#### `harmonized_9trials_reproduced.csv` *(primary, validation aid)*
- Format: CSV, 1,781 rows × 19 cols. Row order matches the template.
- Generated by: `scripts/extract_harmonized_clinical.py`.
- What you get: the pipeline's re-derivation of the 9 template trials
  only. Designed to be diff'd against
  `cross_trial_analysis_egk_april30_meta_9trials.csv` line-by-line.
- When to use: when you want to compare the pipeline output to the
  original template in a side-by-side spreadsheet.

### Validation / audit

#### `validation_summary.csv` *(validation/audit)*
- Format: CSV, 135 rows × 5 cols (9 trials × 15 measurable columns).
- Generated by: `scripts/validate_extractions.py`.
- What you get: per-(trial, column) match rate of the reproduced
  9-trial output against the template. The single most useful overview
  of "where the reproduction is and isn't perfect."
- When to use: early — to know which (trial, column) pairs are at 1.000
  and which warrant a closer look.

#### `validation_report.csv` *(validation/audit)*
- Format: CSV, 3,455 rows × 8 cols (one row per mismatching cell).
- Generated by: `scripts/validate_extractions.py`.
- What you get: every cell-level disagreement between the reproduction
  and the template, with both values shown.
- When to use: after `validation_summary.csv` flags a (trial, column),
  filter this file by that (trial, column) to see the specific rows.

#### `row_order_diagnostics.csv` *(validation/audit)*
- Format: CSV, 1 row × 6 cols when the alignment is perfect.
- Generated by: `scripts/extract_harmonized_clinical.py`.
- What you get: audit of the 9-trial alignment to the template under key
  `(trial, cimac_part_id, Cimac.id, Collection_Event)`. Any non-summary
  row would indicate a missing/extra/duplicate key.
- When to use: confirm `harmonized_9trials_reproduced.csv` is row-by-row
  comparable to the template before doing a diff.

#### `exclusion_and_order_checks.txt` *(validation/audit)*
- Format: plain text with a final `VERDICT:` line.
- Generated by: `scripts/build_exclusion_and_order_checks.py` (last
  wrapper step).
- What you get: a single text file confirming (1) BACCI is excluded,
  (2) the final trial count is 11, (3) the 9-trial reproduction matches
  template row order, (4) no stale `harmonized_12trials.csv`, and (5)
  no unexpected BACCI mentions in any regenerated report.
- When to use: **always glance at this first** — it is a one-shot
  pass/fail for the pipeline's invariants.

#### `template_anomalies.csv` *(validation/audit)*
- Format: CSV, machine-readable list of known template quirks.
- Generated by: `scripts/generate_review_report.py`.
- What you get: data-quality anomalies in the **original 9-trial
  template** that the reproduction faithfully preserves (e.g. the
  EAY131-Z1D `arm` column carries sex values for 628 rows — preserved
  verbatim rather than "fixed"). Useful when the template itself looks
  wrong.
- When to use: when an apparent mismatch in `validation_report.csv` is
  actually a template defect, not a reproduction defect.

### Provenance

#### `provenance_long.csv` *(provenance)*
- Format: CSV, 37,988 rows × 12 cols.
- Generated by: `scripts/extract_harmonized_clinical.py`.
- What you get: one row for every value emitted (or attempted) by the
  pipeline. Columns include `source_file`, `source_column`,
  `source_row_idx`, `extraction_method`, `confidence`, and free-text
  `notes`. **This is the source-of-truth audit trail.**
- When to use: to answer "where did THIS specific value come from?".
  Filter by `(trial, cimac_part_id, Cimac.id, Collection_Event,
  harmonized_field)`.

#### `source_evidence_report.csv` *(provenance)*
- Format: CSV, 209 rows × 11 cols (one row per (trial, harmonized_field)).
- Generated by: `scripts/generate_review_report.py`.
- What you get: a per-(trial, field) summary of which source files and
  columns contributed values, which extraction methods were used, and
  the confidence distribution.
- When to use: when you want a single-row answer to "what does the
  pipeline use to populate field X for trial Y?" rather than scanning
  thousands of provenance rows.

### Human-review

#### `final_handoff_report.md` *(human-review, start here)*
- Format: Markdown narrative + tables.
- Generated by: `scripts/build_final_handoff.py`.
- What you get: an executive narrative — final output files, validation
  summary, intentionally unresolved fields, committed rule assumptions
  (with evidence), tested-but-not-committed hypotheses, trials/fields
  needing human review, and a change log.
- When to use: **start here.** It is written for reviewers.

#### `final_handoff_report.csv` *(human-review)*
- Format: CSV, 187 rows × 9 cols.
- Generated by: `scripts/build_final_handoff.py`.
- What you get: the same content as `final_handoff_report.md` but as a
  flat table: `(trial, field, status, n_rows, match_rate, n_NA,
  n_flagged, …)`. Sort/filter friendly.
- When to use: when you want to slice the handoff by status (`OK`,
  `MOSTLY_OK`, `PARTIAL`, `NEEDS_REVIEW`, `ALL_NA`, `MOSTLY_FLAGGED`,
  `POPULATED_FROM_SOURCE`).

#### `review_priority_checklist.md` *(human-review)*
- Format: Markdown grouped by priority.
- Generated by: `scripts/build_review_checklist.py`.
- What you get: 23 prioritized P1/P2/P3 review items. Each has
  `Issue type`, `Rows affected`, `Current behavior`, `Candidate source
  files / columns`, `Evidence`, `Question for reviewer`, and
  `Recommended next action`.
- When to use: **the operational to-do list** for reviewers.

#### `review_priority_checklist.csv` *(human-review)*
- Format: CSV, 23 rows × 11 cols.
- Generated by: `scripts/build_review_checklist.py`.
- What you get: same content as the MD, sortable.
- When to use: tracking which items have been addressed.

#### `nonperfect_match_review.md` *(human-review)*
- Format: Markdown.
- Generated by: `scripts/build_nonperfect_match_review.py`.
- What you get: every (trial, column) cell with template match rate
  < 1.0 (excluding policy-NA fields), classified by severity, with
  pattern descriptions.
- When to use: when you want to understand why a near-perfect cell
  isn't perfect (rounding, NaN handling, off-by-one, etc.).

#### `gu16257_pfs_time_fallback_investigation.md` *(human-review)*
- Format: Markdown narrative.
- Generated by: `scripts/build_gu16257_pfs_time_investigation.py`.
- What you get: the read-only investigation that motivated the
  committed rule `pfs_time = DFSTIM if not NaN else DRFSTIM` for
  CIMAC-gu16257. Documents the pattern this team uses for committing
  fallback rules with strong evidence.
- When to use: as a *template* for the kind of evidence you should
  expect when committing similar fallback rules elsewhere.

#### `human_review_summary.txt` *(human-review)*
- Format: plain text.
- Generated by: `scripts/generate_review_report.py`.
- What you get: a flat narrative summary of the pipeline run —
  per-trial row counts, flagged-field tallies, tested hypotheses, and
  intentionally unresolved fields.
- When to use: a quick orientation read before opening the MD reports.

#### `flagged_for_review.csv` *(human-review, drill-down)*
- Format: CSV, 7,126 rows × 12 cols.
- Generated by: `scripts/extract_harmonized_clinical.py`.
- What you get: one row per cell whose extraction confidence fell below
  threshold (and was therefore set to NA in the harmonized output).
  Columns include `source_files`, `candidate_source_variables`,
  `observed_source_values`, `proposed_mapping`, `confidence_score`,
  `reason_low_confidence`, `question_for_reviewer`.
- When to use: when you've decided to investigate a specific (trial,
  field) flagged in the checklist or handoff — filter to that pair to
  see all proposed mappings.

### Diagnostic / log

#### `file_inventory.csv` *(diagnostic)*
- Format: CSV.
- Generated by: `scripts/inspect_trial_files.py`.
- What you get: every file in every `*-clinical/` source dir with size
  + sha256.
- When to use: confirming the same source-file set was used as in a
  prior run.

#### `duplicate_files.csv` *(diagnostic)*
- Format: CSV.
- Generated by: `scripts/inspect_trial_files.py`.
- What you get: groups of source files with identical sha256 content.
- When to use: rare — only when a duplicate is suspected.

#### `headers_by_file.json` *(diagnostic)*
- Format: JSON.
- Generated by: `scripts/inspect_trial_files.py`.
- What you get: the column headers (and Excel sheet names) of every
  source file. Useful for column-name spelunking.
- When to use: when a checklist item asks "is column X present in file
  Y?".

#### `inspect_summary.txt` *(diagnostic)*
- Format: plain text.
- Generated by: `scripts/inspect_trial_files.py`.
- What you get: human-readable per-trial source-file summary.
- When to use: orientation; rarely a primary review file.

#### `fresh_output_audit.txt` *(diagnostic, if present)*
- Format: plain text.
- Generated by: previously written read-only audit (not by the wrapper).
- What you get: a structured pass/fail audit of the most recent
  clean wrapper run.
- When to use: confirming the most recent clean re-run passed all
  invariants.

## 3. Suggested review order

The pipeline is built to push you from **high-level summary → priority
list → specific cells → underlying source data**, in that order.

1. **`exclusion_and_order_checks.txt`** *(30 seconds)*
   Look for `VERDICT: PASS`. If it isn't PASS, stop and surface the
   failure before doing anything else.

2. **`final_handoff_report.md`** *(10–15 minutes)*
   Read top-to-bottom. This gives you the validation overview,
   intentionally unresolved fields, committed assumptions (with
   evidence), and a list of trial/field cells that need attention.

3. **`review_priority_checklist.md`** *(20–30 minutes)*
   Walk through P1 items first, then P2, then P3. Each item already
   tells you the exact files and columns to look at.

4. **`nonperfect_match_review.md`** *(10 minutes)*
   Read the severity classification. Many sub-1.000 cells are benign
   (rounding, NaN-vs-NaN); this file says which ones are.

5. **`flagged_for_review.csv`** *(targeted)*
   When the checklist points to a specific (trial, field), filter this
   file by `trial == 'XXX' AND harmonized_field == 'YYY'` to see what
   the pipeline considered and why it abstained.

6. **`validation_report.csv`** *(targeted)*
   For specific row-level disagreements with the template, filter by
   `trial == 'XXX' AND column == 'YYY'` and inspect the
   `template_value` vs `reproduced_value` columns.

7. **`provenance_long.csv`** *(targeted, deep)*
   For the ultimate "where did this exact value come from?" question,
   filter by the 4-column anchor key plus `harmonized_field`. Each row
   names the source file, source column, source row index, and method.

8. **`harmonized_11trials.csv`** *(last)*
   Only after you trust the rules. This is the dataset itself; opening
   it before reviewing the rules invites cherry-picking by eye.

## 4. Human decision map

Each decision below is currently the reason a value, or set of values,
is `NA` (or below the accepted match threshold) in the harmonized
output. Resolving these is the main path to a more complete dataset.

For each: `decision_id`, `priority`, `scope`, `field(s)`, `why`,
`current pipeline behavior`, `files to inspect`, `filters / columns`,
`possible outcomes`, `what to tell Claude`.

---

### D1 — `bor_bin` derivation rule
- **Priority:** P3 (high impact, low urgency until rule confirmed)
- **Scope:** all 9 template trials, ~1,781 rows
- **Field:** `bor_bin`
- **Why a human decision is needed:** the template's `bor_bin` does
  **not** follow `BOR.binary` alone (SD splits 99/178 between
  bor_bin=0 and bor_bin=1). The most likely rule is the standard
  clinical-benefit indicator (R **OR** SD lasting ≥ X days), but the
  exact PFS threshold per trial is unknown.
- **Current pipeline behavior:** `bor_bin` is `NA` for every template
  row and every (trial, row) is in `flagged_for_review.csv` per
  the no-silent-guessing policy.
- **Files to inspect:**
  - `final_handoff_report.md` § "Hypotheses tested but NOT committed"
  - `provenance_long.csv` filtered to `harmonized_field=='BOR.binary'`
    plus `pfs_time`
- **Useful filters:** filter the original template by trial; cross-tab
  `BOR.binary` against `bor_bin` and against `pfs_time` to see where
  the hypothesis breaks.
- **Possible outcomes:**
  - (a) Adopt rule `bor_bin = 1 iff BOR.binary=='R' OR (BOR.binary=='SD'
    AND pfs_time >= N)` with a confirmed threshold `N` (per-trial or
    global).
  - (b) Adopt a different rule from the analysis SAP.
  - (c) Accept the field as permanently `NA`.
- **What to tell Claude (after a decision):**
  > "The approved `bor_bin` rule is: 1 iff BOR.binary=='R' OR
  > (BOR.binary=='SD' AND pfs_time >= 183). Implement this reproducibly
  > through `harmonization_config.yaml` + the extractor base, do not
  > manually edit harmonized CSVs, preserve provenance with a
  > `derived_bor_bin_rule` extraction_method, rerun the wrapper, and
  > summarize before/after match rates."

### D2 — `pfs_bin` derivation rule
- **Priority:** P3
- **Scope:** all 9 template trials, ~1,781 rows
- **Field:** `pfs_bin`
- **Why a human decision is needed:** the template's `pfs_bin` is
  **not** a copy of `pfs_stat`. Hypothesis `1 iff pfs_time>=183 OR
  pfs_stat==0` matches ~81% on s1400i and Z1D — suggestive of a
  6-month PFS landmark indicator but not exact.
- **Current pipeline behavior:** `pfs_bin` is `NA` for every template
  row.
- **Files to inspect:**
  - `final_handoff_report.md` § "Hypotheses tested but NOT committed"
  - `provenance_long.csv` filtered to `harmonized_field` in
    `{'pfs_stat', 'pfs_time'}`.
- **Useful filters:** filter the template by trial; cross-tab
  `pfs_stat` × `pfs_bin` × bins of `pfs_time`.
- **Possible outcomes:**
  - (a) Adopt 6-month PFS landmark rule with a confirmed threshold.
  - (b) Adopt a different SAP-defined rule.
  - (c) Accept the field as permanently `NA`.
- **What to tell Claude:**
  > "The approved `pfs_bin` rule is: 1 iff pfs_time >= N OR
  > pfs_stat == 0. Implement via config + extractor base, preserve
  > provenance, rerun the wrapper, and summarize before/after."

### D3 — S1400I per-sample (decimal) `age`
- **Priority:** P2 (blocks 561 rows for one trial)
- **Scope:** `CIMAC-s1400i`, 561 rows
- **Field:** `age`
- **Why a human decision is needed:** the template stores decimal
  per-sample age at collection. The source file's `age_num` is integer
  age at enrollment. The pipeline cannot derive per-sample age because
  no S1400I source file contains per-sample collection dates.
- **Current pipeline behavior:** `age` is `NA` for all S1400I rows;
  each row is in `flagged_for_review.csv`.
- **Files to inspect:**
  - `S1400I-clinical/` (source files) — confirm no collection-date
    file exists.
  - `provenance_long.csv` filtered to `trial=='CIMAC-s1400i' AND
    harmonized_field=='age'`.
- **Possible outcomes:**
  - (a) Supply a per-sample collection-date file (the pipeline can then
    derive age = enrollment_age + (collection_date − enrollment_date) /
    365.25).
  - (b) Accept enrollment age (`age_num`) as a degraded substitute, and
    document the divergence from the template.
  - (c) Accept the field as `NA` for S1400I.
- **What to tell Claude:**
  > "Supplied S1400I per-sample collection dates in
  > `S1400I-clinical/sample_collection_dates.csv` with columns
  > `cimac_part_id`, `Cimac.id`, `collection_date`. Add a derivation in
  > the S1400I extractor, do not manually edit CSVs, rerun and
  > summarize."

## 5. How to implement human decisions

**Do not** manually edit any CSV in `harmonization_outputs/`. Manual
edits break provenance, fail validation on the next run, and quietly
get overwritten the next time the wrapper runs.

The correct workflow is:

1. Record the decision in a review log (your own notes / a shared
   doc / a ticket). Capture: the decision, the evidence, who approved
   it, the date.
2. Hand the decision to Claude (or whoever maintains the pipeline)
   with a clear instruction. Use this template:

   > **Approved decision for `<field>` in `<trial / scope>`:** `<rule>`
   >
   > **Evidence:** `<one or two sentences pointing to the file(s) and
   > rows that support it>`
   >
   > **Instructions:** implement this rule reproducibly through
   > `scripts/config/harmonization_config.yaml` (preferred) or the
   > relevant extractor in `scripts/extractors/`. Do not manually edit
   > harmonized CSVs. Preserve provenance with a meaningful
   > `extraction_method` and `notes`. Rerun
   > `./scripts/run_full_harmonization.sh` and summarize before/after
   > match rates and any new flagged rows.

3. After Claude / the maintainer reports back: re-run a fresh audit
   (see § 7) to confirm nothing else regressed.

## 6. Minimal vs full audit paths

### Minimal path — "is `harmonized_11trials.csv` usable for my analysis?"
*(~30 minutes)*

1. Open `exclusion_and_order_checks.txt`. Confirm `VERDICT: PASS`.
2. Open `final_handoff_report.md`. Read §1 (file inventory), §2
   (validation summary), §3 (intentionally unresolved).
3. Confirm none of the fields you actually need for your analysis
   are in §3 (unresolved) or §6 (sub-1.0 cells).
4. Open `nonperfect_match_review.md` and confirm the sub-1.0 cells
   that *do* matter are classified as benign.
5. You can use `harmonized_11trials.csv` with the caveat list from §3.

### Full audit path — "I need source-supported confidence for every cell I report"
*(half a day to a day)*

1. Run the minimal path above.
2. Walk through `review_priority_checklist.md` end-to-end. For each
